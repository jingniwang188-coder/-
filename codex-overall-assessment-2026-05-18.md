# 塔罗之眼项目整体评估与下一步行动方案

评估日期：2026-05-18  
评估范围：当前文件夹内源码、接口、配置、既有交接文档、本地预览结果  
结论口径：以当前代码为准，旧文档中部分判断已经被后续开发追上

## 1. 一句话结论

“塔罗之眼”已经不是早期 demo，而是一个主流程完整、体验感较强、具备小范围测试基础的塔罗 Web 产品。它目前最接近的阶段是“可体验 Beta / 可内测验证”，距离正式收费运营还差支付验收、线上环境配置、移动端 QA、安全加固和可维护性整理。

## 2. 当前项目进展

### 2.1 技术形态

- 前端是原生 HTML、CSS、JavaScript 单页应用，没有 React、Vue、Next.js 等框架。
- 页面结构在 `index.html`，主要交互与业务逻辑集中在 `script.js`，视觉与响应式样式集中在 `style.css`。
- 后端接口放在 `api/` 目录，形态是 Vercel Serverless Functions。
- 数据层目前分两类：
  - 用户解读历史、札记、日签缓存：浏览器本地 `localStorage` / `sessionStorage`。
  - VIP 支付订单：PostgreSQL，通过 `pg` 连接。
- 当前 `package.json` 只有 `pg` 依赖，没有本地开发、测试、构建脚本。

### 2.2 已经完成的产品能力

- 首页已经有三个明确入口：
  - 单张神谕
  - 主题占问
  - 双人关系合盘
- 牌库当前已经是完整 78 张塔罗牌：
  - 22 张大阿卡那
  - 权杖、圣杯、宝剑、星币四组小阿卡那各 14 张
- 已配置 8 个牌阵：
  - 单牌神谕
  - 是非决断阵
  - 时间之流
  - 情感透视阵
  - 财富事业阵
  - 月运导航阵
  - 二选一岔路阵
  - 灵感十字阵
- 抽牌体验已经较完整：
  - 开场 splash
  - 时相主题
  - 水晶球互动
  - 洗牌动画
  - 牌扇展开
  - 点击抽牌
  - 逐张翻牌
  - 牌面预览
- AI 解读链路已接入：
  - 前端请求 `/api/tarot`
  - 后端调用 DeepSeek Chat Completions
  - 长文解读支持流式输出
  - 失败时有简版 fallback
- 留存与复盘能力已经有雏形：
  - 成长档案
  - 解读历史
  - 心境札记
  - 今日神谕日签
- 结果导出已经实现：
  - 可生成 9:16 海报图片
  - 目前更偏“保存图片”，不是完整社交分享链路
- 反馈通道已实现：
  - 优先通过 Resend 发邮件
  - 未配置时回退到 mailto
- VIP/付费能力当前比旧文档描述更进一步：
  - 前端已有订单创建、状态查询、轮询、支付后解锁流程
  - 后端已有支付宝预创建订单、支付状态查询、订单核销、支付宝异步回调、管理员手动标记支付成功能力
  - 如果支付宝预创建未配置，可以回退到静态二维码模式

## 3. 网站成熟度判断

### 3.1 产品成熟度：中等偏高

主流程已经成立，且不是简单页面拼接。用户能从首页进入问题、抽牌、翻牌、生成解读、保存记录、回看档案，这是一条真正的产品闭环。

但它还不是“可放心正式收费运营”的状态。原因不是功能缺失，而是关键链路缺少线上验收和运营级防护。

当前更适合：

- 给朋友、小范围用户内测
- 验证塔罗内容风格
- 验证付费意愿
- 收集体验问题

暂不建议直接大规模投放或正式收费推广。

### 3.2 体验成熟度：中等偏高

优点很明显：

- 视觉风格统一，有“塔罗之眼”的独立气质。
- 首屏、本地预览、按钮入口、主页布局都能正常呈现。
- 本地浏览器预览没有发现控制台报错。
- 结果页有摘要、章节折叠、牌名高亮、海报导出，已经超过普通 demo。

风险也比较明显：

- 动效、背景、牌面图、音效、外部脚本较多，弱网和低性能手机上需要专项验证。
- 移动端虽然做了不少适配，但还没有看到系统化测试记录。
- 主要逻辑集中在一个 3500 多行的 `script.js` 中，继续扩展会越来越难控。

### 3.3 技术成熟度：中等

技术实现能跑，功能覆盖也比较完整，但工程化程度偏低：

- 没有测试脚本。
- 没有 lint / format 规则。
- 没有构建流程。
- 没有类型检查。
- 没有端到端测试。
- 关键配置依赖环境变量，但 `.env.example` 不完整。

这不是说项目差，而是说明它还处在“产品原型成长为正式项目”的过渡期。

### 3.4 商业化成熟度：中等偏低到中等

支付能力已经写得比较完整，但还需要正式验收：

- 需要确认线上 `DATABASE_URL` 可用。
- 需要补齐支付宝相关环境变量示例。
- 需要用支付宝沙箱或真实小额订单跑完整链路。
- 需要验证静态二维码回退逻辑是否符合实际运营方式。
- 需要增加订单、金额、产品类型、token 的更严格绑定。

所以它已经有收费产品的骨架，但还没有完成收费产品的验收。

## 4. 优点

### 4.1 核心产品闭环完整

用户不是只看一张牌，而是能经历完整过程：

1. 选择入口
2. 输入问题或进入日签
3. 选择牌阵
4. 抽牌
5. 翻牌
6. AI 解读
7. 存入成长档案
8. 保存海报或继续复盘

这条链路是项目最值得保留的资产。

### 4.2 产品风格统一

“塔罗之眼”的命名、深色星空、仪式感动效、水晶球、牌面、羊皮纸解读、成长档案之间是同一个世界观，不是零散功能堆叠。

### 4.3 当前牌库已经补全

旧报告里曾提到牌库不是完整 78 张，但当前代码已经包含 78 张。对后续产品口径来说，这是一个重要进展，可以更有底气地描述为完整塔罗牌库。

### 4.4 AI 解读 prompt 有方向感

后端 prompt 已经不是单纯“写一段塔罗解读”，而是要求：

- 先给结论
- 给关键提醒
- 解释为什么
- 给现在可执行的建议
- 避免空泛玄学话术

这让产品有机会从“氛围占卜”走向“咨询式辅助决策”。

### 4.5 支付后端基础扎实

已有订单表、支付事件表、支付宝回调验签、金额校验、订单状态、订单核销、管理员手动标记等基础能力。这个方向是对的，后续重点是验收和加固，不是推倒重来。

### 4.6 留存模块已经有雏形

成长档案、历史记录、札记、日签组合在一起，已经提供了用户回来的理由。后续如果加账号和云同步，这部分可以继续长成长期产品价值。

## 5. 缺点与风险

### 5.1 线上配置不完整

`.env.example` 目前只列出少量变量，缺少支付宝、反馈邮件、VIP 口令、朋友测试码等关键配置示例。对于接手、部署和排查问题都不够友好。

建议优先补齐：

- `ALIPAY_APP_ID`
- `ALIPAY_APP_PRIVATE_KEY`
- `ALIPAY_PUBLIC_KEY`
- `ALIPAY_NOTIFY_URL`
- `VIP_UNLOCK_CODE`
- `VIP_FRIEND_TEST_CODES`
- `RESEND_API_KEY`
- `FEEDBACK_TO_EMAIL`
- `FEEDBACK_FROM_EMAIL`

### 5.2 VIP 后端校验存在配置绕过风险

`api/tarot.js` 里高级牌阵校验依赖 `VIP_UNLOCK_CODE` 是否存在。也就是说，如果线上忘记配置 `VIP_UNLOCK_CODE`，高级牌阵可能不会被强制校验 VIP token。

这类逻辑对商业化很敏感，建议改成：只要牌阵需要付费，就必须校验 token；口令配置只影响“口令入口”，不影响后端付费权限判断。

### 5.3 VIP token 和订单绑定还不够强

当前 token 主要包含过期时间和 scope。对于正式收费，更理想的是把 token 与本次订单、产品类型、金额或用途绑定，避免跨场景误用。

建议 token payload 至少加入：

- `orderId`
- `productType`
- `amountFen`
- `iat`
- `exp`

### 5.4 AI 输出渲染需要更严格消毒

前端会对模型输出做一些 HTML 清理，也在 prompt 中要求模型不要输出 HTML，但最终仍通过 `marked.parse()` 写入 `innerHTML`。模型输出不可完全信任，建议引入严格 HTML 白名单或 DOMPurify 一类的清理方案。

### 5.5 外部依赖较多

当前体验依赖多个外部资源：

- Google Fonts
- marked CDN
- html2canvas CDN
- sacred-texts 塔罗牌图片
- Pixabay 音效
- 第三方二维码生成服务

这些都会在弱网、跨域、限流、资源失效时影响体验。正式上线前，应把关键依赖分级：

- 核心依赖尽量本地化或自托管
- 非核心依赖提供降级
- 失败时给用户稳定反馈

### 5.6 工程可维护性开始吃紧

`script.js` 已经超过 3500 行，包含牌库、状态、支付、AI、历史、日签、海报、动画等全部逻辑。短期还能维护，长期继续加功能会变得困难。

暂时不建议大重构成框架项目，但建议做“轻量模块化”：

- 把牌库拆出去
- 把支付逻辑拆出去
- 把历史/札记拆出去
- 把海报生成拆出去
- 把 API service 拆出去

### 5.7 分享能力还不完整

结果页目前有保存图片、查看档案、再问一次。保存图片能满足“带走结果”，但不是完整分享。

如果后续希望自然传播，需要补：

- Web Share API
- 复制链接
- 分享落地页或可恢复的 reading id
- 海报二维码指向真实站点并可配置

### 5.8 成长档案还是单设备能力

历史和札记存本地，优点是轻量，缺点是：

- 换浏览器就没了
- 清缓存就没了
- 不能跨设备
- 无法做用户级长期分析

这不影响内测，但会限制长期留存。

## 6. 下一步行动方案

建议下一阶段目标不要再横向堆功能，而是做一个“可内测收费版”。核心目标是：支付可验收、AI 可稳定、移动端可用、安全边界可接受、后续有人能接手。

## 7. 推荐版本：V1.1 可内测收费版

### 7.1 P0：先做线上可用性与支付验收

目标：让项目可以被真实用户小范围使用，并且付费链路可信。

任务：

1. 补齐 `.env.example`
2. 明确 `OPENAI_API_KEY` 实际是 DeepSeek key，建议改名或增加注释
3. 配置线上 `DATABASE_URL`
4. 配置支付宝沙箱或正式应用参数
5. 跑通一次完整付费链路：
   - 创建订单
   - 生成二维码
   - 用户支付
   - 支付宝回调
   - 订单变为 paid
   - 前端轮询识别成功
   - 订单换 token
   - token 进入 AI 解读
   - token 被消费
6. 验证静态二维码回退模式：
   - 创建订单失败时的用户提示
   - 管理员手动标记支付成功
   - 前端是否能继续轮询到状态变化
7. 修复 VIP 后端校验配置绕过风险
8. 增强 token 与订单/产品类型绑定

验收标准：

- 免费牌阵无需支付可完成解读。
- 进阶牌阵未支付无法通过后端生成解读。
- 已支付订单能自动继续解读。
- 订单过期后不能继续解锁。
- 支付金额不一致不能标记成功。

### 7.2 P0：做一轮核心流程 QA

目标：确认主流程在真实浏览器里稳定。

任务：

1. 桌面端测试：
   - 首页加载
   - 单张神谕
   - 时间之流
   - 进阶牌阵支付弹窗
   - 成长档案
   - 保存海报
   - 反馈弹窗
2. 移动端测试：
   - iPhone Safari
   - Android Chrome
   - 小屏宽度
   - 低网速环境
3. 弱网测试：
   - AI 接口慢
   - 牌面图加载失败
   - marked 未加载
   - html2canvas 加载失败
4. 错误状态测试：
   - 未配置 API key
   - 支付接口不可用
   - 数据库不可用
   - Resend 未配置

验收标准：

- 用户不会卡在无反馈状态。
- 核心按钮都能点击并产生正确状态变化。
- 失败时能看懂发生了什么。
- 浏览器控制台没有关键错误。

### 7.3 P1：稳定 AI 输出与内容安全

目标：让解读内容稳定、可读、安全。

任务：

1. 引入 HTML 消毒方案，替代当前仅靠正则清理。
2. 给 AI 输出增加最小结构校验：
   - 是否有结论
   - 是否有行动建议
   - 是否过短或空白
3. 给 fallback 解读加更明确的用户提示。
4. 把 prompt 中的产品调性沉淀成独立文档，方便后续迭代。

验收标准：

- 模型输出 HTML 或异常格式时不会污染页面。
- 用户最先看到的是结论，而不是铺垫。
- 网络失败时不会丢失抽牌结果。

### 7.4 P1：降低关键外部依赖风险

目标：让核心体验不被第三方静态资源拖垮。

任务：

1. 将 `marked` 改成本地依赖或本地托管。
2. 将 `html2canvas` 改成本地依赖或本地托管。
3. 评估塔罗牌图片是否需要自托管。
4. 为音效加载失败提供静默降级。
5. 二维码生成服务增加备用方案或后端生成。

验收标准：

- CDN 不可用时，核心解读仍可展示。
- 海报生成失败时，用户结果仍保留。
- 牌面图加载失败时，emoji 兜底正常。

### 7.5 P1：轻量模块化

目标：不大重构，但让后续维护更轻。

建议拆分顺序：

1. `tarot-data.js`：牌库、牌面映射、牌阵配置
2. `payment.js`：VIP 订单、轮询、token、弹窗状态
3. `history.js`：成长档案、历史、札记
4. `poster.js`：海报生成
5. `tarot-api.js`：AI 请求、日签请求、fallback 请求

验收标准：

- 拆分后功能不变。
- 首页、抽牌、支付、保存海报都能跑通。
- `script.js` 明显变短，主流程更容易阅读。

### 7.6 P2：补真正分享与长期留存

目标：从“好用”走向“能传播、能复访”。

任务：

1. 增加系统分享：
   - 支持 Web Share API
   - 不支持时复制链接
2. 设计可分享结果页：
   - 生成 reading id
   - 可公开查看摘要或海报
   - 隐私问题默认隐藏
3. 增加云端历史或账号体系：
   - 可以先用轻账号或邮箱 magic link
   - 不必一开始做复杂会员中心
4. 成长档案增强：
   - 问题主题统计
   - 情绪趋势
   - 最近重复课题

验收标准：

- 用户可以一键分享。
- 被分享者能回到产品。
- 老用户换设备后不丢关键记录。

## 8. 建议的执行顺序

### 第一周

- 补齐环境变量说明
- 跑通本地和线上基础部署
- 支付链路沙箱验收
- 修复 VIP 后端强制校验逻辑
- 记录一份支付验收日志

### 第二周

- 做桌面端和移动端核心流程 QA
- 修复首批阻断性问题
- 加 AI 输出消毒
- 加关键错误状态提示
- 固化一份测试清单

### 第三周

- 本地化关键脚本依赖
- 优化海报生成失败兜底
- 开始轻量拆分 `script.js`
- 小范围找 5-10 个真实用户测试

### 第四周

- 根据内测反馈优化文案和流程
- 补系统分享/复制链接
- 决定是否进入正式收费试运营

## 9. 暂时不建议优先做的事

- 不建议马上重写成 React/Vue。
- 不建议马上做复杂账号系统。
- 不建议先做大量新牌阵。
- 不建议先做增长裂变活动。
- 不建议在支付和安全没验收前大规模推广。

现在项目最需要的是把已经很有感觉的主流程打磨到“稳定可信”，而不是继续加更多入口。

## 10. 最终判断

这个项目已经有很好的产品胚子：视觉气质成立，主流程成立，AI 解读方向成立，成长档案和付费骨架也都已经出现。它的问题不是“没做出来”，而是“还没完成正式运营前最后一轮硬化”。

下一步最值得做的是 V1.1 可内测收费版：支付验收、安全加固、移动端 QA、AI 输出稳定、关键依赖降级。做完这一轮后，“塔罗之眼”就可以从一个漂亮可玩的作品，升级成一个可以认真拿出去测试商业化的小产品。
